Use case:

(1). Оформление заказа официантом

Действующее лицо: официант

Цель: оформление заказа

Предусловия:
1. Официант имеет список доступных для заказа позиций.
2. Официант знает свой логин и пароль.
3. Блюдо доступно для заказа.

Триггер: клиент разместил заказ

Основной сценарий:
1. Официант вводит свои учетные данные.
2. Система верифицирует данные и позволяет доступ.
3. Показывается меню.
4. Официант выбирает "Создать новый заказ" из меню.
5. Открывается диалоговое окно с настройками заказа.
6. Официант устанавливает параметры заказа и открывает его.
7. Показывается меню ресторана.
8. Официант добавляет позиции в заказ.
9. Официант подтверждает заказ через меню.
10. Система подтверждает заказ и отправляет на обработку.

Альтернативный сценарий:
1. Повторяются шаги 1-4 из основного сценария.
2. Официант вводит ID клиента.
3. Система предлагает рекомендации по выбору блюд для данного клиента.
4. Продолжение шагов 5-8 из основного сценария.

Постусловие: заказ успешно оформлен.

(2). Предоставление информации о блюде

Действующие лица: официант, клиент

Цель: предоставить детальную информацию о блюде

Предусловия:
1. Официант знаком с блюдом, по которому требуется информация.
2. Официант знает свои учетные данные.
3. Блюдо присутствует в меню.

Триггер: клиент запрашивает дополнительные сведения о блюде

Основной сценарий:
1. Официант аутентифицируется в системе.
2. Система подтверждает его учетные данные.
3. Показывается меню.
4. Официант находит нужное блюдо в поиске.
5. Система предоставляет информацию о блюде.
6. Официант показывает клиенту карточку с информацией о блюде.
7. Клиент получает необходимые данные.

Постусловие: клиент узнал все, что ему было нужно о блюде.

(3). Расчет стоимости заказа и формирование чека

Действующие лица: официант, клиент

Цель: рассчитать стоимость заказа

Предусловие:
1. Существует заказ для конкретного клиента.
2. Официант знает свои учетные данные.
3. У клиента достаточно средств для оплаты.

Триггер: клиент запросил расчет стоимости заказа

Основной сценарий:
1. Официант входит в систему своими учетными данными.
2. Система аутентифицирует его и разрешает доступ.
3. Официант выбирает нужный заказ из меню.
4. Официант выбирает метод оплаты и подтверждает выбор.
5. Система запрашивает подтверждение официанта.
6. Официант подтверждает свой выбор.
7. Система генерирует чек и помещает заказ в статус "ожидает оплаты".
8. Клиент оплачивает заказ.
9. Официант подтверждает оплату.
10. Система подтверждает получение данных.
11. Заказ удаляется из "ожидают оплаты" и переходит в статус "завершен".

Альтернативные сценарии:
1. Повторяются шаги 1-7 из основного сценария.
2. Клиент решает изменить метод оплаты.
3. Официант выбирает нужный заказ из "ожидающих оплаты".
4. Официант меняет метод оплаты и запрашивает новый чек.
5. Повторяются шаги 6-10 из основного сценария.

Постусловие:
1. Создан новый чек в базе данных.
2. Обновлено количество продуктов на складе.
3. При идентифицированном клиенте обновлена сумма его заказов.

Действующие лица: официант, клиент

Цель: занесение данных клиента в базу

Предусловия:

3. Официант знает свои учетные данные.

Триггер: Проявление интереса клиента к программе лояльности

Основной сценарий:
1. Официант вводит свои данные.
2. Система подтверждает доступ.
3. Официант выбирает раздел "Клиенты" в меню.
4. Система показывает список клиентов и функцию редактирования.
5. Официант выбирает "Добавить клиента".
6. Система открывает окно для заполнения данных клиента.
7. Официант заполняет данные и подтверждает.
8. Система подтверждает прием данных.
9. Клиент добавлен в базу данных.

Постусловие: Клиент успешно добавлен в базу данных.

Нефункциональные требования:

1. Приложение должно отличаться интуитивным интерфейсом, доступным для всех пользователей.
2. Приложение должно быть мультиплатформенным, поддерживать запуск на различных ОС.
3. Приложение должно корректно функционировать для небольших заведений, таких как кофейни или кафе.
4. Язык интерфейса приложения должен быть на русском.
5. Цены в приложении должны отображаться в российских рублях.
6. На этапе разработки не планируется интеграция с внешними системами.
7. Вопросы безопасности не требуют особого внимания, так как в системе отсутствуют критические данные.
8. Веб-приложение должно быть совместимо с основными популярными браузерами.
9. Приложение должно демонстрировать стабильную работу при небольших нагрузках для малых или средних заведений, таких как кофейни или кафе.

User story:

В роли официанта, я стремлюсь к лучшей ориентации среди блюд и категорий, чтобы обслуживание клиентов проходило более эффективно.

В роли официанта я хочу иметь возможность идентифицировать клиента при оформлении заказа, чтобы предложить ему необходимые блюда.

В роли официанта, я стремлюсь к доступу к полной информации о каждом блюде, чтобы в случае необходимости предоставить ее клиенту.

В роли официанта, важно иметь возможность выбора способа оплаты, чтобы не вызывать неудобства у клиентов.

В роли официанта, важна возможность быстрого редактирования заказов для более эффективной работы.

В роли официанта, знание списка заказов поможет мне оценить объем работы, выполненной за определенный период.

В роли официанта, важно знать время окончания смены, чтобы планировать свою работу.

В работе официанта, доступ к информации о предыдущих заказах клиентов поможет мне оказать помощь в выборе блюд.

В роли официанта, важно уметь рассчитывать окончательную стоимость заказа и составлять чеки для обслуживания клиентов.
